## Why this wave exists

Chat is the first migrated feature whose state is not "fetch a
payload, render it". It has:

- a reducer (`chat-reducer.ts`) modelling thread state (idle / sending
  / streaming / error),
- a hook (`use-chat.ts`) wrapping the reducer with API calls,
- a screen with five sibling components
  (`chat-screen`, `composer`, `message-bubble`, `context-pill`,
  `suggested-prompts`, `chat-empty`) all needing GRACE markers,
- existing legacy unit tests in `legacy/frontend/lib/reducers/
  reducers.test.ts` covering the reducer's transitions.

Three things have to be decided up front:

1. **Streaming.** Legacy `lib/api/chat.ts` returns a fixture; there is
   no real streaming. Subsequent backend wave will introduce SSE or
   AI SDK `streamText`. The wave migrates the reducer **with the
   streaming branches in place but inert** so the wiring is not
   redone later.

2. **Test gate timing.** The reducer carries unit tests in legacy
   (`reducers.test.ts`). W-2.8 ships the JS test gate. Two options:
   (i) bring the tests over now under `lib/grace/` but skip them
   in CI until W-2.8 lights up the gate, or (ii) hold the tests in
   `legacy/` and migrate them in W-2.8. Either way the reducer
   itself moves now.

3. **Reducer placement.** Legacy keeps reducers under `lib/reducers/`.
   The whitelist allows `lib/grace/**`. New canonical home is
   `lib/grace/chat/reducer.ts` (per-feature folder), not a flat
   `lib/grace/reducers/`. Each feature owns its own reducer; the
   onboarding reducer follows the same layout in W-2.7.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Migrate reducer + hook + screen now; keep tests in legacy
  until W-2.8.** The reducer file lands in `lib/grace/chat/
  reducer.ts`. Its unit tests stay in `legacy/frontend/lib/reducers/
  reducers.test.ts` (read-only). W-2.8 ports the test file and
  rewires it. Pro: smallest diff for this wave. Con: live reducer
  has zero direct test coverage between W-2.4 and W-2.8.

- **B. Migrate reducer + hook + screen + reducer tests now; tests
  are inert until W-2.8 lights the gate.** Tests live in
  `lib/grace/chat/reducer.test.ts`, conform to GRACE markers,
  but no runner picks them up yet (vitest config arrives in W-2.8).
  Pro: tests follow the code. Con: dead-looking files in the tree
  until W-2.8.

- **C. Promote W-2.8 forward.** Ship the JS test gate (vitest +
  config + bootstrap Gate 11) before chat. Then this wave reuses it.
  This means the controller swaps W-2.4 ↔ W-2.8 ordering.

The remainder of this packet is conditional on the decision. Default
expectation is **Option B** (tests follow code, gate lights up later).

  `legacy/frontend/lib/contracts/chat.ts` that W-2.1 did not put
  into the chosen contract source. Stop, raise a W-2.1-followup;
  do not redeclare locally.

  feature not yet present (e.g. server-side timestamps with
  timezone). Stub the timestamp using `Date.now()` **inside the
  hook**, never inside the reducer; record the stub under
  `## Decision`.
- Vitest is brought forward (Option C) but a transitive dependency
  conflicts with current devDeps. Resolve in W-2.8 packet, not
  here.
- The composer relies on a shadcn primitive missing from
  `components/ui/**`. Add via the shadcn CLI (`pnpm dlx shadcn@
  latest add <name>`); record the addition.
