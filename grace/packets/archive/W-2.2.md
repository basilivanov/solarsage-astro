## Why this wave exists

W-2.0 installed the rails. W-2.1 reconciled the type sources. W-2.2 is
the first wave that actually puts a working UI inside the GRACE
perimeter. It migrates two things together because they are coupled at
the layout boundary:

1. **Shell.** `legacy/frontend/components/app-shell.tsx` plus
   `legacy/frontend/app/(app)/layout.tsx` â€” bottom nav, page chrome,
   client providers (`telegram-init`, `theme-provider`), trial banner.
   No subsequent feature page renders without a shell, so the shell
   has to land first.
2. **Today.** `legacy/frontend/app/(app)/page.tsx` is a redirect to
   today; the actual screen is `legacy/frontend/components/today/
   today-screen.tsx` plus 7 sibling components, plus the API layer
   `legacy/frontend/lib/api/today.ts`, `lib/api/config.ts`, `lib/api/
   index.ts`, plus the data fixtures `lib/api/today.fixtures.ts`.

Splitting Today into a "components only" wave and a "wiring" wave is
not done here because the W-2.0 marker gate requires every file under
the whitelist to carry MODULE_CONTRACT and FUNCTION_CONTRACT blocks. A
component without its API layer is non-renderable, so the gate would
flag a half-migrated state as a phantom module.

This wave also performs the first replacement of the W-2.0 stub at
`app/(grace)/today/page.tsx` with a real, rendering page. The stub was
deliberately kept as a rendering example for the marker gate; this
wave deletes it and replaces it.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Single route group `(grace)`.** Move the shell to
  `app/(grace)/layout.tsx` and the page to `app/(grace)/today/page.tsx`
  (already exists as W-2.0 stub; replaced here). All future migrated
  pages also live under `(grace)`. Pro: matches the W-2.0 whitelist
  exactly, no whitelist edit needed. Con: route group name carries
  meta-process information into the URL tree; if it is ever surfaced
  in a route segment config it becomes coupling.

- **B. Functional route group `(app)` mirroring legacy.** Move the
  shell to `app/(app)/layout.tsx`, the page to `app/(app)/today/
  page.tsx`. Update `grace/frontend.paths` whitelist to add
  `app/(app)/**`. Pro: zero diff against legacy structure, easier
  diff review during W-2.3..W-2.7. Con: whitelist edit; the meta-name
  `(grace)` survives only as a dead route group; W-2.0 pilot has to
  move.

- **C. Two-tier: `(grace)/(app)/today/page.tsx`.** Outer `(grace)`
  marks the GRACE perimeter, inner `(app)` carries the shell. Pro:
  encodes both axes. Con: Next.js route groups stack but do not
  multiply layouts; this becomes confusing fast.

The remainder of this packet is conditional on the decision. The
worker uses only the subset of allowed paths matching the chosen
option. Default expectation in this packet is **Option A**.
