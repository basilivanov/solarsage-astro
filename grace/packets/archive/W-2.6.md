## Why this wave exists

This wave consolidates everything that depends on **user identity and
access state**:

- **Profile screen.** 8 components: avatar, profile-row, edit-sheet,
  service-row, access-card, horary-card, referral-card, dev-mode-
  switcher.
- **Paywall.** Modal-style screen shown when access state = `expired`
  or `none`.
- **Trial banner.** Shell-level banner shown when access state =
  `trial`. The W-2.2 shell ships this with a stubbed `useAccess()`
  returning a constant; this wave replaces the stub.
- **Telegram-init.** Reads Telegram WebApp launch params, populates
  user identity (`useTelegramUser`).
- **Onboarded gate.** `useOnboarded()` decides whether to mount the
  shell normally or redirect to onboarding (W-2.7). This wave wires
  the gate but **not** the onboarding flow itself.

Three architectural concerns:

1. **Access is shell-coupled.** `app/(grace)/layout.tsx` reads
   `useAccess()` to decide whether to render the trial banner.
   Migrating access without simultaneously migrating the shell
   consumer leaves the W-2.2 stub in place; that stub is then a
   silent fallback. To prevent that, this wave **edits**
   `app/(grace)/layout.tsx` to switch from stub to real hook.
   This is the only wave permitted to edit the shell after W-2.2.

2. **Telegram-init runs before the rest.** It must mount before any
   hook that reads identity. Order is enforced by component nesting
   in the layout, not by global state.

3. **Cities API is shared.** The profile edit-sheet uses cities for
   birthplace selection; onboarding (W-2.7) reuses the same client.
   This wave introduces `lib/api/cities.ts` and `lib/grace/cities/
   *` so W-2.7 has it ready.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Migrate everything in one wave.** Profile screen, paywall,
  trial banner real wiring, telegram-init real wiring, cities API,
  all access plumbing. Pro: one consistent state of "shell now sees
  real access". Con: largest single wave in the W-2.x line.

- **B. Two sub-waves.** W-2.6a: access + telegram-init + trial
  banner real wiring + paywall (everything that touches the shell).
  W-2.6b: profile screen + cities (everything that doesn't). Pro:
  smaller diffs, paywall behaviour testable independently. Con:
  cities API is a dependency of W-2.7 (onboarding); if W-2.6b slips,
  W-2.7 stalls.

- **C. Defer paywall to a separate wave.** This wave migrates
  profile + access + telegram-init + cities + trial banner; paywall
  ships in W-2.6P. Pro: paywall has its own product-level concerns
  (pricing, copy, subscription routing) that may want product
  iteration. Con: mid-state where access reports `expired` but no
  paywall is shown.

The remainder of this packet is conditional on the decision. Default
expectation is **Option A** (one consistent shell state).
