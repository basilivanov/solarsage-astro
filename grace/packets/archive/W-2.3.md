## Why this wave exists

Calendar and Day are coupled at the URL boundary: tapping a cell in
`/calendar` navigates to `/day/[date]`. Splitting them across two
waves causes either (a) a calendar that links nowhere and breaks
e2e flow, or (b) a day route reachable only by hand-typed URL.

Day is also the **first wave with a dynamic segment**
(`day/[date]/page.tsx`). The W-2.0 marker gate and W-2.0 lint rules
have not yet been exercised against:

- a route file that uses `params: { date: string }`,
- a server-side date validator (the route must reject malformed
  date keys before the screen renders),
- a `cacheLife`/`revalidateTag` decision (Day is a per-date page,
  Today is a singleton).

Recording these decisions explicitly is part of what this wave does.

The Calendar screen also depends on `DayStatusMap`, which is one of
the domains W-2.1 reconciled. If W-2.1 chose Option A (backend-driven),
the calendar wires through `packages/contracts.CalendarPayload`. If
W-2.1 chose Option B/C, it imports from `lib/grace/contracts/calendar`.
Worker reads W-2.1 `## Decision` before writing imports.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. CSR for both screens.** Both calendar grid and day reading are
  client components. Data fetched via SWR keyed by month / date. Pro:
  matches legacy implementation byte-for-byte; lowest risk. Con:
  every navigation triggers a JS fetch; no prerender for known dates.

- **B. RSC for `/day/[date]` (cacheable), CSR for `/calendar`.** Day is
  a per-date page; status is determined by ephemeris (server-side
  pure function); response is cacheable with `'use cache'` + a
  `cacheLife('days')` profile. Calendar stays CSR because the user
  scrubs months and we want instant grid updates. Pro: free SSR HTML
  for shared `/day/...` links; no extra client JS for day reading
  body. Con: requires a server data path that does not exist in
  legacy; introduces `'use cache'` (Next 16).

- **C. RSC for both, with client islands.** Same as B but `/calendar`
  is also RSC; month scrubbing happens via shallow router updates and
  a client-only `MonthScrubber` island. Pro: smallest JS bundle. Con:
  most invasive vs legacy; month transitions are visibly slower.

The remainder of this packet is conditional on the decision. The
worker uses only the subset of allowed paths matching the chosen
option. Default expectation in this packet is **Option A** (parity
with legacy); architectural upgrades are deferred to a later wave.

## Allowed Write Scope (assumes Option A; deltas for B/C noted)

### Routing

- `app/(grace)/calendar/page.tsx` — module M-WEB-CALENDAR. Renders
  `<CalendarScreen />`.
- `app/(grace)/day/[date]/page.tsx` — module M-WEB-DAY. Validates
  `params.date` against `^\d{4}-\d{2}-\d{2}$` and 404s on mismatch
  (Next.js `notFound()` from `next/navigation`). Renders
  `<DayReading date={...} />`.
- Under Option B/C: `app/(grace)/day/[date]/page.tsx` becomes a
  server component with `'use cache'` + `cacheLife('days')`; the
  client component `DayReading` is imported from
  `components/grace/today/day-reading.tsx` (already migrated in
  W-2.2).

### Components

- `components/grace/calendar/calendar-screen.tsx` — port of legacy
  `components/calendar/calendar-screen.tsx`.
- `components/grace/calendar/mood-icon.tsx` — port of legacy
  `components/calendar/mood-icon.tsx`. Maps `DayStatus` (`tense` /
  `even` / `supportive`) to icon + color token.

### API client + fixtures

- `lib/api/calendar.ts` — port of legacy. Imports `CalendarPayload`
  / `DayStatus` / `DayStatusMap` only from `packages/contracts`
  (Option A in W-2.1) or from `lib/grace/contracts/calendar`
  (Option B/C in W-2.1).
- `lib/api/calendar.fixtures.ts` — port of legacy mock; must
  type-check against the chosen contract source.
- `lib/api/index.ts` — append calendar exports to the existing
  barrel from W-2.2. No new file.

### Pure helpers (whitelist already covers `lib/grace/**`)

- `lib/grace/calendar/month-grid.ts` — port of legacy `lib/calendar.ts`.
  Pure function `buildMonthGrid(year: number, month: number,
  statusMap: DayStatusMap)` returning the grid model used by the
  screen. Module M-WEB-CALENDAR-PURE.
- `lib/grace/date.ts` — port of legacy `lib/date.ts`. Pure date
  helpers used by both calendar and day. Module M-WEB-DATE-UTIL.

### Whitelist + canon

- `grace/frontend.paths` — no edit; existing globs cover all paths
  above.
- `docs/GRACE_CANON.md` — no edit.

### Tests

- No JS tests in this wave (gate ships in W-2.8). Pure helpers in
  `lib/grace/**` are designed to be unit-testable; tests are added
  in W-2.8 against the same files.

  - profile / paywall / trial / telegram-init (W-2.6)
  - onboarding (W-2.7)

    in which case escalate first)

  legacy reuses it; no second variant.
