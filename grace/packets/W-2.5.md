## Why this wave exists

Readings is two routes (`/readings`, `/readings/natal`) and a dynamic
sub-route (`/readings/natal/[section]`). The natal page renders a
`NatalSection` whose body is a discriminated union of blocks
(paragraph, bullets, planets-widget, spheres-widget, highlights-strip,
…). The block renderer is a switch on `block.kind` that dispatches to
widget components.

Three coupled architectural concerns live here:

1. **Block renderer = lint-fragile.** A switch over a discriminated
   union is the canonical pattern that the W-2.0 lint rule
   `contracts-only-import` aims to protect: the union must come from
   `packages/contracts` (or `lib/grace/contracts/natal` per W-2.1),
   never be redeclared. Every widget receives its block subtype via
   narrowing, never via casting.

2. **In-development overlay.** Legacy `components/readings/in-dev-
   overlay.tsx` is shown over readings that exist as items but are not
   yet implemented. This is product-level state, not a contract field,
   so it lands as a UI-only flag inside the readings screen. The
   wave records explicitly which readings render their content vs.
   show the overlay.

3. **Section-level routing.** `/readings/natal/[section]` is the
   second dynamic-segment route after W-2.3's `/day/[date]`. The
   `section` param is enumerated (`identity`, `relations`, `career`,
   …); validation rejects unknown values with `notFound()`. This
   is enforced by a required marker block, mirroring the
   `DATE_PARAM_VALIDATION` block from W-2.3.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Migrate everything: `/readings`, `/readings/natal`,
  `/readings/natal/[section]`, plus all widgets, in one wave.** Pro:
  legacy parity; no orphan routes. Con: largest single wave; biggest
  diff.

- **B. Two sub-waves, both inside W-2.5.** W-2.5a: index +
  natal landing (no `[section]` route, no widgets). W-2.5b: section
  route + widgets. Pro: smaller diffs. Con: between W-2.5a and W-2.5b,
  natal landing links to a 404; user-visible breakage during the gap.

- **C. Migrate index + natal landing + section route, but **stub**
  widgets to placeholder cards. Real widgets land in a follow-up
  micro-wave W-2.5W.** Pro: section routing is testable; widget
  authoring is decoupled. Con: visible regression from legacy until
  W-2.5W ships.

The remainder of this packet is conditional on the decision. Default
expectation is **Option A** (legacy parity) given that the widgets are
already content-only (no new data dependencies).

## Must Preserve

- INV-MARKER-COMPLETE (Gate 9 = 0). Every page and component carries
  the full marker set.
- INV-CONTRACTS-ONLY-IMPORT (Gate 8 = 0).
- INV-DRIFT-CLEAN.
- INV-LEGACY-INERT.
- INV-NO-CASTS-IN-RENDERER: `block-renderer.tsx` does not contain
  `as ` casts or `any`. Verifiable by:
  - the file's `MODULE_CONTRACT.invariants` listing this rule, and
  - a grep gate (a small line in `scripts/grace/check-markers.sh`
    or a sibling script) flagging ` as ` and `: any` inside the
    file. Worker may add this grep as part of this wave **only** as
    a file-scoped append; no global lint rule is introduced.
- INV-PARAM-VALIDATION: section route validates `params.section`
  before render, mirroring W-2.3 day route.
- INV-WIDGET-PROPS-FROM-CONTRACT: each widget's props type is a
  named slice from the natal block union, not a freshly declared
  interface. Verifiable by reviewer diff.

## Escalation triggers

- A legacy block kind is missing from `NatalBlock` union in the
  contract source. Stop, raise W-2.1-followup; do not add the
  variant locally.
- Legacy widget references a third-party charting lib not
  installed. Stop and decide: install via `pnpm add` (record under
  `## Decision`) or fall back to a shadcn-only rendering.
- The renderer cannot be expressed without a single narrowing
  helper (`assertNever`). That helper is canonical and lives at
  `lib/grace/util/assert-never.ts` with full markers; record its
  introduction under `## Decision`.
- The in-dev overlay condition (`isInDev`) needs to be data-driven
  per reading. Stop and raise a W-2.1-followup to add a boolean
  field to `ReadingBody`; do not encode it in the UI.
