## Why this wave exists

W-1.2/1.3/1.4 stood up real auth, `/api/day/:date`, and
`/api/calendar?month=YYYY-MM`. The frontend has been on
`NEXT_PUBLIC_USE_FIXTURES=true` since W-1.1B, importing typed shapes
from `packages/contracts/*` but resolving every payload from
`lib/api/*.fixtures.ts`. This wave flips the switch: when
`NEXT_PUBLIC_USE_FIXTURES=false` (or unset), `lib/api/*` calls the
real backend.

This wave is the demo gate for PHASE-1 and the entry-criterion for
W-1.6 (logging spine retrofit). It is **only** a data-source switch
and a thin error-state pass. No new UI components, no design changes,
no new pages, no new contracts. The visual output for happy paths
must be byte-identical to fixtures mode (modulo `meta.generated_at`
timestamps).

PHASE-2 (W-2.x) is *not* gated on W-1.5; W-2.0 already merged its
gate-rail. But W-2.2 (the first real legacy port) needs `lib/api/*`
to actually call the backend, otherwise the ported page renders
nothing. So W-1.5 is the seam between "the API exists" and "the UI
uses it".

## Decision required from controller (record under `## Decision`)

Pick exactly one fetch layer.

- **A. Fetch + tiny wrapper.** `lib/api/_client.ts` exports
  `apiFetch<T>(path, init?) → Promise<T>` built on `fetch`,
  `credentials: "include"`, base URL from `NEXT_PUBLIC_API_BASE`,
  errors normalized to `ApiError {code, message, status}`. Per-domain
  modules (`lib/api/today.ts`, `lib/api/calendar.ts`,
  `lib/api/auth.ts`, `lib/api/profile.ts`) call `apiFetch`. Pro:
  zero deps; small surface; aligns with how `legacy/frontend/lib/api/*`
  is already organized; correlation header (W-1.6) plugs in trivially
  in one place. Con: no built-in caching; SWR-on-top added in a later
  wave if needed.

- **B. SWR client now.** Same wrapper, but `lib/api/use*.ts` hooks
  expose SWR. Pro: cache + revalidation for free; matches v0 default
  guideline. Con: scope creep — this wave is supposed to be a switch,
  not a state-management refactor; the legacy frontend uses RSC fetch,
  not SWR, so a port becomes incidental.

- **C. Fetch wrapper now, RSC server fetch where applicable.** For
  pages that the legacy port lands as Server Components, use server-
  side `fetch` with cookie-forwarding via `next/headers`. For client
  islands, use the wrapper from Option A. Pro: matches legacy split.
  Con: cookie-forwarding-from-headers is a known footgun (auth cookie
  must be re-sent server-side; correlation header behavior differs).

**Default recommendation: A.** SWR can be added in a later
non-blocking wave. The pilot demo doesn't need cache. RSC vs CSR is a
W-2.2+ concern when actual pages move.

## Fixture cleanup (mandatory, part of this wave)

`lib/api/*.fixtures.ts` files are **deleted** in this wave. The flag 
`NEXT_PUBLIC_USE_FIXTURES` is **removed**. After W-1.5, the frontend always 
calls the real backend.

Rationale:

- Two parallel code paths (fixture vs real API) create maintenance burden.
- Fixtures must be kept in sync with the real API (easy to drift).
- The flag is a footgun (easy to accidentally deploy with fixtures enabled).
- If a backend endpoint is missing, the frontend should **fail loudly** (404 
  or 500), not silently fall back to fixtures. This forces backend and frontend 
  to stay in sync.
- For local development without backend, use the mock backend server from 
  `tests/e2e/fixtures/server.ts` (see "e2e backend strategy" below).

Migration path:

```typescript
// BEFORE (W-1.4 and earlier):
import { USE_FIXTURES } from "./_flags";
import { getDayReal } from "./today.real";
import { getDayFixture } from "./today.fixtures";
export const getDay = USE_FIXTURES ? getDayFixture : getDayReal;

// AFTER (W-1.5):
import { apiFetch } from "./_client";
import type { DayPayload } from "@/packages/contracts";

export async function getDay(date: string): Promise<DayPayload> {
  return apiFetch<DayPayload>(`/api/day/${date}`);
}
// If backend is down → fetch throws → error boundary catches → explicit error UI
```

For local development without backend:

```bash
# Start mock backend server (reuses fixture JSON)
node tests/e2e/fixtures/server.ts &
export NEXT_PUBLIC_API_BASE="http://localhost:3001"
pnpm dev
```

## Error states (only these three; no more)

For both `/day/[date]` and `/calendar` pages:

1. **Unauthenticated (401)**. The wrapper throws
   `ApiError{status:401, code:"UNAUTHENTICATED"}`. Page renders an
   inline "Sign in with Telegram" CTA (using the existing component
   if present in `legacy/frontend/components/auth/*`; **do not port
   it in this wave** — the CTA is a placeholder anchor link that
   reads "Sign in" and sends the user to a `/login` route that
   already exists or, if it does not, to `/`). The acceptance is
   that the page does not white-screen.
2. **Validation error (422)**. The wrapper extracts `code` and
   `message`; the page renders an inline panel: "Bad request:
   {code}". Reload button.
3. **Server error (5xx) or network error**. The wrapper throws
   `ApiError{status:0|5xx, code:"NETWORK"|"SERVER"}`; the page
   renders an inline panel: "Couldn't reach server". Reload button.

These three are the *only* error UIs introduced in this wave. A real
error boundary is W-2.x material.

## e2e backend strategy

Playwright e2e specs target three modes, registered as projects in
`playwright.config.ts`:

- `fixtures-only`: `NEXT_PUBLIC_USE_FIXTURES=true`. Asserts visual
  parity (snapshot of `/day/today` and `/calendar`).
- `mock-backend`: `NEXT_PUBLIC_USE_FIXTURES=false`. A tiny in-process
  mock under `tests/e2e/fixtures/server.ts` answers
  `/api/auth/telegram`, `/api/profile`, `/api/day/:date`,
  `/api/calendar`. The same JSON fixtures are reused. This mode
  proves the wrapper actually issues HTTP calls with the right
  shape and headers.
- `error-states`: `NEXT_PUBLIC_USE_FIXTURES=false` but the mock
  responds with 401 / 422 / 500 to specific routes; pages must
  render the corresponding inline panel.

Why a mock instead of the real backend in CI: this wave should not
require booting `apps/api` in CI; that becomes a separate
infra-level decision. Locally, developers can run the real backend
and re-run the same specs against it (`PW_BASE_URL` override).

## Business events emitted (declared; routed through `log.event`
after W-1.6)

Per canon §8.5 (frontend events; emitted via `console.error`
placeholder in this wave):

- `ui.api_call_started` — every `apiFetch` call.
- `ui.api_call_succeeded` — 2xx response.
- `ui.api_call_failed` — non-2xx or thrown.
- `ui.error_panel_shown` — when 401/422/5xx panel renders. Payload
  carries `{path, status, code}`.

If any of these names is not in §8.5, raise an amendment to canon
**before** writing the emission code. The emission shape (`name`,
`payload`) is the canon; the channel (`console.error` now,
`log.event` after W-1.6) is not.

## Escalation triggers

- A page renders incorrectly because a field is missing from
  `TodayPayload` / `CalendarPayload`. The fix is in W-1.3/W-1.4
  (Pydantic), not here. Stop, raise a Pydantic extension PR, then
  resume W-1.5.
- The legacy fetch layer for `/day` or `/calendar` includes headers
  or auth flows not yet implemented in W-1.2 (e.g. a custom
  `Authorization` header instead of the cookie). Stop. Either
  amend W-1.2 to issue the expected token shape or accept that the
  page UX in W-1.5 is "cookie + Telegram only".
- Playwright cannot run in the CI environment. Land the unit tests
  and the spec files anyway; mark the e2e step as `continue-on-error`
  for one PR cycle and open a follow-up to fix CI infra. Do not
  delete the specs.
- The drift gate flips red because the wrapper's `ApiError` shape
  leaks into `packages/contracts`. It must not. `ApiError` is a
  **frontend-internal** type; never re-exported through
  `packages/contracts`.
