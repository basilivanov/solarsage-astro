# W-6.2 — Subscription ledger entries + enforcement

## Decision

On successful payment, create subscription entry in access_ledger with 30 days access. Enforcement happens via existing AccessService.can_access_day().

## Acceptance Criteria

- [x] PaymentService.handle_webhook() creates subscription on success
- [x] AccessService.grant_subscription() creates ledger entry
- [x] Subscription entry has days_granted=30, start_date, end_date
- [x] Successful payment → subscription → access granted
- [x] Failed payment → no subscription
- [x] Tests: subscription creation, access enforcement

- File: `/opt/solarsage-astro/apps/api/app/services/access_service.py` — grant_subscription method (already existed from W-ACCESS.1)

## Implementation Details

### PaymentService.handle_webhook()

When payment status is "succeeded":
1. Update payment.completed_at
2. Create AccessService instance
3. Call grant_subscription(user_id, start_date=today, days=30)
4. Commit transaction

### AccessService.grant_subscription()

Already implemented in W-ACCESS.1:
- Creates AccessLedger entry with entry_type="subscription"
- Sets start_date and end_date (30 days)
- Commits to database

### Access Enforcement

Existing AccessService.can_access_day() automatically checks access_ledger entries:
- Subscription entries grant access for their date range
- Returns state="full" when day is covered by subscription
- Returns state="locked" for days outside subscription range

## Tests

1. **test_successful_payment_creates_subscription**: Verifies subscription entry created on payment success
2. **test_failed_payment_no_subscription**: Verifies no subscription on payment failure
3. **test_subscription_grants_access**: Verifies subscription covers today and future days (15 days ahead)

## Negative Tests

- [ ] Duplicate webhook → idempotent (no duplicate subscription)
- [ ] Expired subscription → access denied
- [ ] Subscription for 0 days → validation error
- [ ] Payment without user → graceful failure

## Notes

- Subscription starts from date.today() (payment completion date)
- 30 days is hardcoded for MVP (W-6.1 scope)
- Future waves may add:
  - Variable subscription durations
  - Subscription renewal logic
  - Webhook idempotency
