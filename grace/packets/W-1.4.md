## Why this wave exists

`legacy/frontend/components/calendar/*` renders a 3-month grid:
previous, current, next. It needs one API call per month tab. The
frontend cannot leave fixtures mode (W-1.5) until both `day` and
`calendar` endpoints exist on the real backend, and PHASE-2's locked-
day UI cannot render badges without a calendar payload that carries
`access` per day.

This wave introduces `GET /api/calendar?month=YYYY-MM` returning a
`CalendarPayload`. Per-day `access` comes from the same stub used by
W-1.3 (`state="full"` for all dates). PHASE-2 W-2.1 replaces the stub;
this wave only requires the *shape* and the per-day plumbing.

What this wave does **not** do: no real status computation, no LLM,
no caches, no UI changes, no auth changes. The calendar payload is
fixture-shaped per day with neutral statuses.

## Decision required from controller (record under `## Decision`)

Pick one shape for `dayStatus` in this wave.

- **A. Neutral-only.** Every day in the response carries
  `status="neutral"`, `dayQuality=null`, no top flags. Pro: cheapest;
  matches plan §W-1.4 ("neutral statuses"). Con: the frontend renders
  a visually-flat calendar that says nothing about real days; PHASE-4
  is the first wave where any color appears.

- **B. Reuse W-1.3 fixture's day-level status.** For every day in the
  month, call `today_service.build_payload(...)` and project a small
  subset (`status`, `dayQuality`, `topFlags[:2]`). Pro: visually
  richer demo; same source of truth as `/api/day`. Con: 31 calls per
  month — fixture parse cost matters; cohesion across `day` and
  `calendar` becomes implicit.

- **C. Deterministic synthesis from date.** Compute `status` and
  `dayQuality` from a hash of `YYYY-MM-DD` modulo 5; no fixture call.
  Pro: O(1) per day; no coupling. Con: status on `/api/day/X` and
  `/api/calendar?month=…` for the same X may diverge unless the same
  hash is used in both; that coupling has to be explicit.

**Default recommendation: A**, matching the plan literally. B/C make
the demo prettier but introduce coupling that PHASE-4 will rewrite
anyway. A means every status pixel changes once when real scoring
lands in PHASE-4 — which is the right place to change it.

## Endpoint contract

### `GET /api/calendar?month=YYYY-MM`

- Query param `month`: `YYYY-MM`. No path param.
- Auth: required.
- 200: `CalendarPayload`:
  ```
  CalendarPayload {
    month: "YYYY-MM",
    days: CalendarDay[],
    meta: CalendarMeta,
  }
  CalendarDay {
    date: "YYYY-MM-DD",
    status: "neutral" | "favorable" | "challenging" | "mixed" | "rest",
    dayQuality: number | null,           // 0..1, null in Option A
    topFlags: TopFlag[] | null,          // null in Option A; ≤2 if present
    access: AccessDecision,              // {state:"full"} from stub
  }
  CalendarMeta {
    contract_version: string,            // generator constant
    generated_at: datetime,              // UTC ISO8601
    fixture: "neutral" | "synthesized" | "from_today",
    cached: false,                       // always false in PHASE-1
  }
  ```
- 401: `AuthError` from W-1.2.
- 422: `code in {"INVALID_DATE","INVALID_DATE_FORMAT"}`.
- 403: `code="NOT_ONBOARDED"` when profile has no `birth_data`. Same
  rule as `/api/day` to keep the two endpoints consistent.

`status` literal union must be exact and exhaustive: any other value
is a contract bug. `dayQuality` is `null` in Option A; `0..1` float in
B/C.

## Path-shape locking

The query parameter is `month`, not a path segment. The legacy
frontend (`legacy/frontend/components/calendar/*`) will pass a
`URLSearchParams` and expect `?month=YYYY-MM`. Do not change to a
path segment without amending the legacy reference; this is the
contract surface frozen here.

## Negative tests (must fail before fix, pass after)

- **N-AUTH**: no session cookie → 401, no service call.
- **N-FORMAT**: `month=2026/05`, `month=2026-5`, `month=` → 422
  `INVALID_DATE_FORMAT`.
- **N-DOMAIN**: `month=1899-12`, `month=2101-01` → 422 `INVALID_DATE`.
- **N-DAY-COUNT**: build the response for 2024-02 → exactly 29 days;
  for 2025-02 → exactly 28; for 2024-04 → exactly 30; for 2024-12 →
  exactly 31. Single parametrized test, four cases.
- **N-ACCESS-PROPAGATION**: monkeypatch `access_service` per-day;
  exact match in response.
- **N-NOT-ONBOARDED**: profile without birth_data → 403, no
  `build_calendar` call.

## Escalation triggers

- A field referenced by `legacy/frontend/components/calendar/*` is
  missing from `packages/contracts/calendar.ts` after this wave
  (e.g. `weekday`, `isToday`). Add it as a Pydantic field with a
  default; do not patch the frontend.
- The legacy frontend expects `month` as a path segment, not a query
  parameter. Stop and decide whether to amend the legacy reference
  or change the route. The route shape is the contract surface;
  pick one and document it.
- DST / timezone math turns into a special case (it should not, since
  every `date` is naive UTC for calendar purposes). If a real
  timezone enters the picture, this is a contract amendment, not a
  fix.
- Option C drifts the `today` snapshot from W-1.3. Stop. The shared
  synth must be used by both surfaces or by neither.
