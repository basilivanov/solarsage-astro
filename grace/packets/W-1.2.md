## Why this wave exists

W-1.1 stood up a FastAPI shell with `/api/health` and an empty alembic
baseline. W-1.1B fixed the contract pipeline on Option B: Pydantic is the
single source of truth, `packages/contracts/_generated.ts` is regenerated
from FastAPI's `openapi.json` and protected by the drift gate.

There is no concept of a user yet. Every later wave depends on one:

- W-1.3 needs a `current_user` to scope `GET /api/day/:date` and to feed
  `user_id_hash` into the canon §8.4 redactor.
- W-1.4 same for `/api/calendar`.
- PHASE-2 access wave reads from a `users` row.
- PHASE-6 payments need a stable `user_id` to attach a subscription.

This wave introduces the smallest amount of identity that unblocks all
of them: Telegram WebApp `initData` HMAC verification, a `users` table,
a `user_profiles` table, an opaque session cookie, and read/write of the
profile resource. No frontend integration, no caches, no referral logic,
no payments. The frontend keeps running on
`NEXT_PUBLIC_USE_FIXTURES=true` and is **not** modified.

## Decision required from controller (record under `## Decision`)

Pick exactly one option for the session mechanism.

- **A. Opaque session cookie + server-side store.** `POST /api/auth/telegram`
  validates `initData`, upserts `users`, mints a random 32-byte token,
  writes a `sessions` row (`user_id`, `token_hash`, `expires_at`,
  `created_at`), sets `Set-Cookie: session=<token>; HttpOnly; Secure;
  SameSite=Lax; Path=/; Max-Age=…`. Logout deletes the row. Pro: server
  controls invalidation; rotation is trivial; no JWT key management.
  Con: one DB read per authenticated request (mitigated by an in-memory
  TTL cache in W-3.x).

- **B. Stateless signed JWT in cookie.** Same flow but the cookie carries
  a JWT signed with `GRACE_SESSION_SECRET`. No DB write for sessions.
  Pro: zero session DB load. Con: invalidation requires a denylist
  table (i.e. server-side state anyway), key rotation is harder, and
  every PHASE-2+ rule that depends on "kick this user now" needs that
  denylist.

- **C. Telegram-only re-validation per request.** No session at all;
  every authenticated request re-verifies `initData` from a header.
  Pro: stateless and aligned with how Telegram WebApps issue `initData`
  per launch. Con: `initData` has a `auth_date` that ages out within
  hours; web-only users (not in TG WebApp) cannot use the API; HMAC
  cost on every hot path.

**Default recommendation: A.** Option B re-introduces server state via
the denylist anyway; Option C breaks the assumption (held by W-1.5
e2e and by every subsequent wave) that a logged-in browser session
survives across page navigations and reloads.

The remainder of this packet is conditional on the decision. Allowed
Write Scope below covers the union; worker uses only the subset
matching the chosen option.

## Endpoint contract

### `POST /api/auth/telegram`

- Body: `TelegramAuthRequest`
- 200: body `AuthSession`, header
  `Set-Cookie: <SESSION_COOKIE_NAME>=<token>; HttpOnly; Secure;
   SameSite=Lax; Path=/; Max-Age=<SESSION_TTL_SECONDS>`
- 400: `AuthError(code=INVALID_HMAC|MISSING_FIELDS)`
- 401: `AuthError(code=INITDATA_EXPIRED)` when
  `now - auth_date > INITDATA_MAX_AGE_SECONDS`
- Side effects: upsert into `users` keyed by `tg_user_id`; insert
  empty `user_profiles` row if absent.

### `POST /api/auth/logout`

- 204; deletes the session row (Option A) / writes denylist (Option B)
  / no-op (Option C).
- Idempotent: missing/expired cookie still returns 204.

### `GET /api/profile`

- Auth required. 200 → `ProfileRead`. 404 → `AuthError(code=
  PROFILE_NOT_FOUND)` only if `user_profiles` row was somehow not
  created (should not happen post-auth; covered by a regression test).

### `PUT /api/profile`

- Auth required. Body: `ProfileWrite`. 200 → `ProfileRead`. Updates
  `updated_at`. Calls `profile_service.invalidate_caches(user_id)` —
  in this wave the function is a no-op marker emitting a `print` line
  (or `log.event` after W-1.6 retrofit).

## HMAC verification (canonical Telegram WebApp recipe)

`telegram_auth.verify(init_data: str, bot_token: str, max_age_s: int)
 -> TelegramUser` does, in order:

1. Parse `init_data` as a query string.
2. Extract `hash`. If absent → `MISSING_FIELDS`.
3. Build `data_check_string`: every key except `hash`, sorted
   alphabetically, joined as `key=value` lines.
4. Compute `secret_key = HMAC_SHA256(key="WebAppData", data=bot_token)`.
5. Compute `computed_hash = HMAC_SHA256(key=secret_key,
   data=data_check_string)`.
6. `hmac.compare_digest(computed_hash.hex(), hash)`. If false →
   `INVALID_HMAC`.
7. `auth_date` integer. If `now - auth_date > max_age_s` →
   `INITDATA_EXPIRED`.
8. Parse `user` JSON; require `id`. Return.

Constant-time compare is mandatory. Raw `init_data` and `hash` must
not appear in any log statement (canon §8.4 redact-keys; this wave
emits via `print()` for now and lets W-1.6 retrofit take care of the
envelope, but raw values in messages are still forbidden — write the
literal string `"<redacted>"` in any log site that touches them).

## Decision

**Option A — Opaque session cookie + server-side `sessions` store.**
Decided 2026-05-25.

Rationale: Option B's only advantage (zero session-table reads) collapses
the moment W-3.x or PHASE-6 needs forced invalidation, because the
denylist that B requires for "kick this user now" re-introduces server
state with worse properties than a `sessions` table — every read becomes
a denylist lookup instead of a primary-key fetch, and key rotation forces
a fleet-wide JTI walk. Option C breaks the session-survival assumption
held by W-1.5 e2e (session must outlive a page reload) and excludes
non-Telegram-WebApp callers entirely; we'd have to reinvent A on top of
C the moment we add a web fallback. Option A is also cheapest to
mitigate later — a TTL cache layered in W-3.x turns the per-request
DB read into the same O(1) the denylist would be, without changing the
contract.

Locked consequences:

- The Allowed Write Scope subset is **Common ∪ Option-A-only**:
  - `apps/api/app/db/models.py` adds `sessions` (no `session_denylist`,
    no `core/jwt.py`).
  - `apps/api/alembic/versions/0001_users.py` includes `users`,
    `user_profiles`, `sessions` in a single revision.
  - `current_user` dependency reads the cookie, hashes it (sha256),
    looks the row up by `token_hash`, asserts `expires_at > now()` and
    `revoked_at IS NULL`. No HMAC re-verification per request.
  - Logout deletes (or marks `revoked_at`) the session row; cookie is
    cleared with `Max-Age=0`. Idempotent.
- `GRACE_SESSION_SECRET` is **not** an env var in this wave (Option B
  only). Token entropy comes from `secrets.token_bytes(32)`.
- The follow-on amendment for in-memory TTL cache is owned by W-3.x,
  not by this wave; do not anticipate it here.
- API/env conventions (route shapes `GET /api/day/{date}`, env names
  `API_BASE_INTERNAL` server-only, `NEXT_PUBLIC_API_BASE` browser-only
  if needed) are inherited from canon and **first manifested** in this
  wave's `.env.example` and `apps/api/app/main.py` router mounts.
  No frontend files are touched (INV-NO-FE-CHANGE).
