## Why this wave exists

This wave has two goals:

### Goal 1: Retrofit existing call-sites (moved from W-1.6)

After W-1.6, the logging spine exists but only NEW code uses it. Existing 
`print()` / `console.log()` call-sites in W-1.1 … W-1.5 deliverables still 
use ad-hoc logging. This creates two parallel logging styles.

This goal retrofits all existing call-sites to use `log.event()`, making 
gate-13 (envelope shape) enforceable on the whole repo.

### Goal 2: Frontend → backend log shipping (original W-1.7 scope)

After W-1.6, frontend in production buffers up to 200 envelopes in
memory and never sends them anywhere. Without shipping, we have no
field visibility into UI errors, no correlation with backend
requests after the fact, and the `ui.*` events in §8.5 are
effectively dead.

The fix is small but has its own concerns:

1. **Backpressure / flooding.** A buggy client could emit thousands
   of events per second. We need batching, rate-limiting, and a
   kill-switch.
2. **Privacy on intake.** Frontend redactor is best-effort (a
   compromised client could send raw PII). The backend MUST re-run
   the same redactor on every received envelope before forwarding to
   stdout.
3. **Schema validation.** Malformed envelopes must be rejected, not
   silently logged, otherwise gate-13 (envelope shape) becomes
   meaningless.
4. **Authentication.** Endpoint must be session-bound. An anonymous
   POST `/api/_log` is a free amplification path.

This wave handles all four. It does not introduce log retention,
search, or external sinks (those are PHASE-6+).

## Decision required from controller

Pick exactly one option and write it under `## Decision` below.

- **A. Stateless forwarding only.** `/api/_log` validates → redacts →
  forwards to stdout. No DB table. Pro: minimum surface; matches
  philosophy "stdout is the log". Con: if an operator wants to query
  client-side errors later, they must have shipped stdout to a sink
  by then.
- **B. Forward + persist last 24h.** Same as A plus a `client_log`
  table with TTL cleanup. Pro: queryable in dev / pilot. Con: new
  table, new migration, new alembic op, retention policy decision
  creeps in.

**Default recommendation: A.** Stdout is the canonical sink. PHASE-6
introduces real sinks (Sentry / Datadog / etc.). A `client_log` table
in PHASE-1 would be a half-built sink that we'd then have to delete.

## Decision

**A** — stateless forwarding. Confirmed by controller.

3. After retrofit, gate-13 (envelope shape) is enforced on the **whole**
   repo, not just new files. Update gate-13 scope in bootstrap.sh.

### Part 2: Frontend → backend log shipping (original W-1.7 scope)

### Endpoint contract

4. `POST /api/_log` accepts JSON body
   `{ envelopes: Envelope[] }` with `1 ≤ len ≤ 50`. Larger payloads
   → 413 Payload Too Large.
5. Each envelope validates against §8.2 schema. First invalid
   envelope short-circuits the batch with 400 and a per-index
   error array. The valid envelopes that came **before** the
   invalid one are still forwarded (best-effort). This is recorded
   as `log.intake_partial` warn-level event with reject reasons
   array.
6. Endpoint requires authenticated session (cookie set by W-1.2
   auth). Anonymous → 401. No bearer-token / api-key flow.
7. Response is `{ accepted: number, rejected: number, errors: [{
   index: number, reason: string }] }` on success or partial.

### Privacy + envelope guarantee

8. Backend re-runs `core/redactor.py` (canon §8.4) on every accepted
   envelope before forwarding. Trust client redactor for nothing.
9. Backend overwrites `correlation_id` field only if absent in the
   envelope. If present, leaves it (correlation across stack).
10. Backend overwrites `received_at` timestamp; trusts envelope's
    `ts` (client clock) but tags `client_clock_skew_ms` if drift > 5s.
11. Backend assigns `user_id_hash` from session, overwriting any
    client-supplied value. Source of truth = server session, not
    client claim.

### Rate limiting + kill-switch

9. Per-`user_id_hash` rate limit: 200 envelopes/min, sliding window.
   Excess → 429 with `Retry-After`. Counted via
   `log.intake_rate_limited` (warn).
10. Global kill-switch: env var `GRACE_LOG_SHIPPING=on|off`. When
    `off`, endpoint returns 503 immediately with body
    `{ disabled: true }`. Default in dev: `off`. Default in prod:
    `on`. Documented in `.env.example`.
11. Frontend shipper respects `503 disabled` by suspending for 60s
    before next attempt.

### Frontend shipper

12. Batches envelopes from the ring buffer (W-1.6) when **either**
    50 envelopes accumulated **or** 5 seconds elapsed since first
    envelope in current batch. Whichever first.
13. POSTs the batch with `X-Correlation-Id` of the **first** envelope
    in the batch (debug aid).
14. On 429: backoff `Retry-After` seconds, then resume.
15. On 5xx: exponential backoff starting 2s, max 60s, max 5 retries
    per batch; on final failure drops the batch and emits
    `log.intake_dropped` (error) into local console only (would loop
    if shipped).
16. On 401: stops shipping for the rest of the page lifetime; user
    is anonymous, the spine continues to log to console / buffer.
17. Hard cap: ring buffer never exceeds 1000 envelopes; oldest
    dropped with `log.buffer_overflow` (warn) emitted **once** per
    overflow streak.

### Tests

18. Unit (backend, 6 tests):
    - happy path 1 envelope
    - happy path 50 envelopes
    - 51 envelopes → 413
    - malformed envelope at index 3 → 400 with errors array,
      indices 0..2 forwarded
    - anonymous request → 401
    - kill-switch off → 503
19. Unit (backend, 1 test): redactor re-applied — sentinel `note`
    in incoming envelope must be `[redacted]` in stdout.
20. Unit (backend, 1 test): rate-limit triggers at envelope 201
    within 60s window for same user.
21. Unit (frontend, 4 tests):
    - batches by size
    - batches by time
    - kill-switch suspension
    - 5xx exponential backoff
22. e2e: a deliberate client-side error in the web app produces a
    log line on the backend with the same `correlation_id` within
    10s. Asserted in `tests/e2e/log_shipping_round_trip.test.ts`.
23. Load test fixture `tests/load/log_shipping.k6.js`: 1000
    envelopes/min sustained for 5 minutes, < 0.1% drops, < 200ms
    p99 latency on `/api/_log`.
