## Why this wave exists

Onboarding is a single screen at `/onboarding` that drives a 5-step
flow (`welcome → birthday → birth → place → done`). It depends on:

- the **onboarding reducer** (legacy `lib/reducers/onboarding-
  reducer.ts`) — pure state machine over `{ step, name, birthday,
  birthTime, city, ... }` with transitions `next`, `back`, `setField`,
  `submit`,
- the **cities API** (already migrated in W-2.6 as `lib/api/
  cities.ts`),
- a **persistence side effect** at `submit`: writes the "onboarded"
  flag (legacy `lib/storage-keys.ts` → `localStorage`) consumed by
  W-2.6's `useOnboarded()`,
- the W-2.6 placeholder route `/onboarding-placeholder` is **deleted**
  here and replaced by the real `/onboarding` route.

Three architectural concerns:

1. **State machine size.** Five steps, but transitions are not
   strictly linear: `birthday` and `birth` both edit the same domain
   object, `place` writes a `city` object resolved via cities API,
   `done` commits. The reducer enumerates the legal transitions; the
   shell hook (`use-onboarding.ts`) calls API and finalizes.

2. **Storage as a declared side effect.** Submit writes a flag to
   `localStorage`. Per W-2.6 escalation note, storage writes go
   through `lib/grace/storage/*` adapters with explicit
   `MODULE_CONTRACT.side_effects: ["localStorage"]`. This wave
   **introduces** that adapter (or confirms W-2.6 introduced it).

3. **Reducer purity vs. step components.** Step components are
   "controlled" — they receive `state` and `dispatch` from the hook
   and never own state. This is the key reason the migration
   carries low regression risk: the dumbest step component is the
   easiest to lint.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Migrate everything: 5 steps + flow + reducer + storage
  adapter, in one wave.** Pro: consistent state. Con: largest
  reducer-bearing wave; if W-2.4 chose Option A (reducer tests in
  legacy), the same choice is applied here for consistency.

- **B. Two sub-waves.** W-2.7a: reducer + flow + storage adapter +
  steps `welcome` and `done` (the two trivial ones). W-2.7b: steps
  `birthday`, `birth`, `place` (the data-heavy ones). Pro: smaller
  diffs; reducer can be exercised end-to-end through trivial steps
  before real steps land. Con: between W-2.7a and W-2.7b the flow
  is incomplete (welcome → done with no data collected).

- **C. Migrate steps and flow now; defer reducer to W-2.8 (when JS
  test gate ships).** Steps use a temporary `useState`-based shell
  hook; reducer migration moves to W-2.8 alongside its tests. Pro:
  reducer ships green-on-tests on day one. Con: temporary hook is
  throwaway code; large rewrite at W-2.8.

The remainder of this packet is conditional on the decision. Default
expectation is **Option A**, mirroring the W-2.4 chat reducer
strategy.

  only path here under `## Decision`, and queue the backend
  endpoint for a future backend wave.

  from `components/ui/**`. Add via `pnpm dlx shadcn@latest add
  <name>`; record the addition.

  randomUUID()` for an id. That's a side effect; relocate id
  generation into the hook (`use-onboarding.ts`) and pass the id
  in via the action payload, keeping reducer pure.
