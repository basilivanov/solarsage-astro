## Why this wave exists

This is the pilot slice for PHASE-1: a real backend endpoint that
returns a real `TodayPayload`, but populated from JSON fixtures rather
than from real ephemerides + scoring + LLM. It exists to:

1. Lock the contract boundary on a real handler (not just a Pydantic
   declaration). The drift gate from W-1.1B then protects every later
   change.
2. Give W-1.5 a real URL to point the frontend at and W-1.6 a real
   call-site to emit `day.*` business events from.
3. Force the seam between `today_service` (pure, fixture-backed here,
   real later) and `access_service` (stub here returning `state=full`,
   real in PHASE-2). Both seams are required by the dependency graph;
   skipping them means W-2.x cannot be authored without rewriting
   handlers.

What this wave does **not** do: no auth changes, no profile changes,
no calendar endpoint, no real scoring, no LLM, no caches, no UI. The
frontend stays on `NEXT_PUBLIC_USE_FIXTURES=true`.

## Decision required from controller (record under `## Decision`)

Pick one fixture-mapping strategy.

- **A. Static fixture per date.** Files
  `apps/api/app/fixtures/day_<YYYY-MM-DD>.json` are committed for a
  small explicit set of dates (≥ 7 covering the pilot demo window).
  For any other date, `day_default.json` is returned with
  `meta.fixture="default"`. Pro: deterministic, easy to review. Con:
  payload diversity is limited to the committed set.

- **B. Deterministic synthesis from date.** A single
  `day_template.json` is shipped; `today_service` mutates it
  deterministically per date (e.g. `dayQuality` derived from a hash of
  `YYYY-MM-DD` modulo 5). Pro: every date renders something
  plausible-looking; e2e can pick any date. Con: harder to assert
  "exact body equals X" in tests; failure modes hide in the synthesis
  function.

- **C. Hybrid.** Static fixtures for the demo set; deterministic
  synthesis for everything else; explicit `meta.fixture="static"` vs
  `"synthesized"` in the response so tests can branch. Pro: best of
  both. Con: two code paths, both to maintain.

**Default recommendation: C.** A is too thin for W-1.5 e2e (which
opens "today" — a moving target); B makes review of the canonical
shape harder; C costs one additional `meta` field that we're already
free to extend.

## Endpoint contract

### `GET /api/day/{date}`

- Path param `date`: `YYYY-MM-DD`. Validation in route via
  `pydantic.types.date` plus an additional clamp to the domain range.
- Auth: required (session cookie from W-1.2).
- 200: `TodayPayload`. Shape per `packages/contracts/today.ts`.
  `meta.cached: false`, `meta.generatedAt: <UTC ISO8601>`,
  `meta.fixture: "static"|"synthesized"`,
  `meta.contract_version: <string>` (constant from generator).
- 401: standard auth error from W-1.2 (`AuthError`).
- 422: `{ "code": "INVALID_DATE_FORMAT" | "INVALID_DATE",
  "message": str }`.
- 403: when the user is not onboarded (no `birth_data`). `code=
  "NOT_ONBOARDED"`. The pilot demo user is onboarded by a fixture user
  in tests; production flow handled by PHASE-3 onboarding wave.

The handler shape:

```
@router.get("/api/day/{date}", response_model=TodayPayload)
async def get_day(
    date: date,
    user: User = Depends(current_user),
    profile: ProfileRead = Depends(current_profile),
    access: AccessService = Depends(),
    today: TodayService = Depends(),
):
    if not profile.birth_data:
        raise NotOnboardedError()
    decision = await access.can_access_day(user.id, date)  # stub: full
    payload = await today.build_payload(user, profile, date, decision)
    return payload
```

`today_service.build_payload` is a pure function: no DB writes, no
network calls. It loads the fixture (Option A), synthesizes (B), or
both (C), patches in the requested `date` and `meta.*`, and returns.

## p95 latency budget

100 sequential `GET /api/day/2026-05-24` calls in
`tests/test_day_endpoint.py::test_p95_latency` must keep p95 under
500ms on the CI runner. The test:

- Warms with 5 calls.
- Times next 100 calls.
- Sorts and reads `times[94]` (p95).
- Asserts `< 0.5 seconds`.

Failure modes worth catching: accidental DB hit per call (the stub
should not query users at all beyond the dependency resolution),
fixture re-parsing on every call (caches the parsed dict at module
load), per-call Pydantic v2 model `model_validate` recursion blow-up.

## Negative tests (must fail before fix, pass after)

- **N-AUTH**: missing session cookie → 401, no fixture read.
- **N-FORMAT**: malformed date in path → 422 `INVALID_DATE_FORMAT`,
  no fixture read.
- **N-DOMAIN**: out-of-range date → 422 `INVALID_DATE`, no fixture
  read.
- **N-NOT-ONBOARDED**: user without `birth_data` → 403, no payload
  built.
- **N-CONTRACT-DRIFT**: artificially mutate `meta.contract_version`
  in `today_service.build_payload` → drift test (`test_today_contract`)
  must fail with a clear field-path message.
- **N-FIXTURE-MARK-ABSENT**: remove `meta.fixture` from response →
  `test_happy_path_2026_05_24` must fail.

## Business events emitted (declared; will route through `log.event`
after W-1.6 retrofit)

Per canon §8.5:

- `day.viewed` — emitted on a successful 200 response. Payload:

any of these names is not in §8.5, raise an amendment to canon

## Escalation triggers

- A field referenced by `legacy/frontend/components/today/*` is
  missing from `packages/contracts/today.ts`. Stop. Raise a separate
  Pydantic extension PR (still inside W-1.3 scope, but recorded
  explicitly under Evidence) instead of patching the frontend.
- The fixture cannot be made to satisfy `extra="forbid"` Pydantic
  validation. The fixture is wrong; do not loosen the model.
- p95 budget cannot be met. Stop and profile; the most likely cause
  is per-call fixture re-parse or per-call Pydantic re-build of the
  whole tree.
- `access_service` stub starts to look like it needs DB. It must
  not. The stub is a constant; if a real DB call is required to
  ship, that's W-2.1 territory and W-1.3 stops being a pilot.
