## Why this wave exists

PHASE-1 currently emits logs ad-hoc: `print()`, FastAPI default access
logs, browser `console.log`. There is no shared envelope, no correlation
between a browser click and a backend response, and no enforced redaction
of `note` / `email` / `tg_user_id` / `birth_*` / payment / auth keys.

PHASE-2 (access, payments, sidecar workers, scoring) cannot ship without
this. Each PHASE-2 wave declares **business events emitted** in its
packet (drift-protection rule, plan §drift-protection). Without W-1.6
there is nothing to emit them through, and gate-11 cannot enforce
registry conformance.

This wave does **only** logging infrastructure plus a mechanical
retrofit of already-merged endpoints to use it. It does not introduce
new endpoints, new routes, new DB tables, new contracts, or new UI.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Retrofit all existing instrumentation in this wave.** Replace
  every `print` / `console.log` / FastAPI access-log line in W-1.1 …
  W-1.5 deliverables with `log.event()` calls in this wave. Pro: by
  end of W-1.6 every line in PHASE-1 conforms to canon §8.2 envelope;
  PHASE-2 starts from a clean baseline; gate-13 is enforceable on the
  whole repo. Con: largest write-scope of any logging-related wave;
  touches files outside `apps/api/app/core/` and `lib/log/`; high
  merge conflict risk.

- **B. Ship the spine only, leave existing call-sites unchanged.**
  W-1.6 lands logger + middleware + codegen + gates wired to *new*
  call-sites only. Existing `print` / FastAPI access-logs stay. Pro:
  smaller diff. Con: gate-13 (envelope shape) cannot be enforced on
  the whole repo because legacy lines fail it; we ship two parallel
  logging styles into PHASE-2; the rule "every line is a canon
  envelope" never becomes true.

- **C. Incremental retrofit (two sub-waves).** W-1.6 = spine + codegen + 
  gates + NEW call-sites only. W-1.7 = retrofit EXISTING call-sites. 
  Pro: smaller individual reviews; lower merge conflict risk; spine can 
  be tested with new code before touching existing code; if W-1.7 slips, 
  PHASE-2 can still start (new code uses spine, old code uses print). 
  Con: between W-1.6 and W-1.7, gate-13 must be in warn-only mode OR 
  scoped to new files only.

**Default recommendation: C (incremental).** Option A has high merge 
conflict risk (touches 12+ files in one wave). Option B leaves gate-13 
permanently fictional. Option C allows spine to be tested with new code 
first, then retrofit existing code incrementally. If W-1.7 slips, PHASE-2 
can still start (new endpoints use spine, old endpoints keep print until 
retrofit).

## Decision

**C (incremental retrofit)** — spine + codegen + gates in W-1.6, retrofit 
existing call-sites in W-1.7.

Rationale: Option A (retrofit all in W-1.6) has high merge conflict risk 
when touching 12+ files. Option C allows the spine to be tested with new 
code first, then retrofit existing code incrementally in W-1.7. Gate-13 
is scoped to new files only in W-1.6, then enforced on whole repo after 
W-1.7. If W-1.7 slips, PHASE-2 can still start (new endpoints use spine, 
old endpoints keep print until retrofit).

**Note:** This changes the original Decision A. Acceptance criteria 18-19 
(retrofit existing call-sites) are moved to W-1.7.

## Acceptance criteria (must all be green to close W-1.6)

### Envelope (canon §8.2)

1. Backend logger writes JSON envelope to stdout in prod-mode and a
   pretty colorized form to stderr in dev-mode. Mode switched by
   `GRACE_LOG_FORMAT=json|pretty` (default: `pretty` if `NODE_ENV` /
   `ENV` ≠ `production`).
2. Every JSON line on stdout parses and validates against the §8.2
   envelope schema. Enforced by gate-13 over the test-suite log
   capture.
3. Frontend logger emits the same envelope to `console.*` (dev) or
   pushes into an in-memory ring buffer of last 200 envelopes (prod;
   shipping is W-1.7).

### Correlation (canon §8.2.correlation_id)

4. Middleware `correlation.py` reads `X-Correlation-Id` from incoming
   request; if absent, mints `corr-{ulid}`; binds it into a
   `ContextVar` for the request lifetime; echoes it in response
   header.
5. Frontend `lib/api/*` injects `X-Correlation-Id` on every fetch,
   reusing the value already on the page (one correlation_id per
   browser navigation; rotates on hard reload).
6. e2e: opening `/day/today` produces ≥1 backend log line with the
   same `correlation_id` as the browser request that initiated it.
   Asserted in `tests/e2e/correlation_round_trip.test.ts`.

### Privacy redactor (canon §8.4)

7. `core/redactor.py` and `lib/log/redactor.ts` are byte-equivalent
   in behaviour: same redact-keys list, same regex patterns, same
   sentinel `[redacted]` / `[redacted-email]` / `[redacted-jwt]` /
   `[redacted-bearer]` / `[redacted-tg-id]` / `[redacted-ipv4]`.
8. Canary suite: for **every** key in §8.4 redact-keys (50 keys), a
   test injects sentinel value `__CANARY_<key>__` into a log call,
   captures the serialized output, and asserts the sentinel does
   **not** appear. Same for every regex pattern (5 patterns).
9. Allow-list keys (§8.4 allow-keys) pass through unchanged. Test
   asserts `correlation_id`, `user_id_hash`, `packet_id` are visible
   in output.
10. Redactor is applied **after** envelope assembly and **before**
    serialization. Asserted by a test that wraps a redact-key in
    `extra={...}` and another that puts it in the message itself.

### Events registry (canon §8.5)

11. Codegen `scripts/codegen/events_from_canon.py` reads
    `grace/canon/observability.xml` and writes
    `apps/api/app/core/logging_events.py` containing a
    `Literal["..."]` type (or `enum.StrEnum`) with all 26 event
    names + payload TypedDicts.
12. Codegen `scripts/codegen/events_from_canon.ts` does the same for
    `lib/log/events.gen.ts` (TypeScript union + payload interfaces).
13. Both codegen outputs are committed to the repo. CI re-runs codegen
    and fails on diff (gate-11 drift). Workflow:
    `.github/workflows/observability.yml`.
14. `log.event(name, payload)` accepts only `name: EventName`.
    mypy / tsc must reject `log.event("typo.bad", {})`.

### Gates wired in CI (canon §8.8)

15. **gate-11 event-registry** — drift check between canon and
    generated artifacts; runs on every PR; blocking.
16. **gate-12 redactor-canary** — runs canary suite (item 8); on
    every PR; blocking.
17. **gate-13 envelope-shape** — collects every log line emitted
    during pytest + jest run; validates each against §8.2 schema;
    blocks if any line is malformed. In W-1.6, this gate is scoped to 
    NEW files only (files created after W-1.6). After W-1.7 (retrofit), 
    gate-13 is enforced over the **whole** repo.

### Retrofit of existing call-sites (MOVED TO W-1.7)

18. **Deferred to W-1.7.** Retrofit of existing `print()` / `console.log()` 
    call-sites in W-1.1 … W-1.5 deliverables is moved to W-1.7 to reduce 
    merge conflict risk and allow spine to be tested with new code first.
    
    W-1.6 scope: spine + codegen + gates + NEW call-sites only.
    W-1.7 scope: retrofit EXISTING call-sites (12+ files).

### Documentation

19. `apps/api/app/core/logging.py` has a module-level docstring
    pointing at canon §8 with examples of `log.event`,
    `log.info/warn/error`, and `with_correlation()` context manager.
20. `lib/log/index.ts` mirrors the docstring as a JSDoc block.
21. README section `### Logging` added to `apps/api/README.md`
    and root `README.md` covering: how to read logs in dev, how to
    add a new event (canon → codegen → emit), kill-switches.

## Business events emitted in this wave

This wave only retrofits already-existing instrumentation onto the
canon envelope. It does not introduce new business events. The
following events from §8.5 begin emitting through the canon envelope
as a side-effect of the retrofit:

- `system.startup`, `system.shutdown`, `system.request`,
  `system.error`, `system.config_loaded` (logger itself; owner W-1.6)
- `auth.tg_login_succeeded`, `auth.tg_login_failed`,
  `auth.session_expired`, `auth.logout` (already-implemented in W-1.2
  endpoints; emission channel changes from `print` to `log.event`)
- `day.viewed`, `day.fixtures_loaded` (W-1.3 retrofit)
- `calendar.viewed` (W-1.3 retrofit)
- `ui.error_boundary_tripped` (W-1.6 owner; first time emitted)

No event added or removed from registry. Any drift = gate-11 fails.
