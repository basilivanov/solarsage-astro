## Why this wave exists

Chat is the most requested feature and the backend is already complete
(W-CHAT-1, W-CHAT-3, W-CHAT-4). This wave migrates the chat UI foundation:
screen layout, message list with virtualization, message bubbles, and empty
state.

Three architectural concerns:

1. **Message virtualization.** Chat history can be long (100+ messages).
   We need virtualized scrolling to maintain performance. Use
   `react-window` or `react-virtuoso` for message list rendering.

2. **Real-time updates.** Messages arrive asynchronously. The UI must
   handle optimistic updates (show user message immediately) and
   streaming responses (show assistant message as it arrives).

3. **State management.** Chat state (threads, messages, loading states)
   lives in Redux (`chatSlice` from W-2.5). This wave connects the UI
   to the existing slice.

  W-10.1c.

expectation is **Option A** (full migration).

## Technical Design

### Components Architecture

```
ChatScreen
├── ThreadList (future W-10.3)
├── MessageList
│   ├── Virtuoso (react-virtuoso)
│   └── MessageBubble (repeated)
│       ├── Avatar
│       ├── Content (markdown)
│       └── Timestamp
├── Composer (future W-10.2)
└── ChatEmpty (when no messages)
```

### State Flow

```
User action → dispatch(sendMessage) → optimistic update → API call
→ backend response → update Redux → re-render
```

### API Endpoints (from W-CHAT-1)

- `GET /api/chat/threads` — list all threads
- `GET /api/chat/threads/:id` — get thread details
- `GET /api/chat/threads/:id/messages` — get messages for thread
- `POST /api/chat/threads/:id/messages` — send message

### Redux State (extend W-2.5)

```typescript
interface ChatState {
  threads: Thread[]
  currentThreadId: string | null
  messages: Record<string, Message[]>
  loading: boolean
  error: string | null
}
```

### Message Schema

```typescript
interface Message {
  id: string
  threadId: string
  role: 'user' | 'assistant'
  content: string
  createdAt: string
  status?: 'pending' | 'sent' | 'error'
}
```
