## Why this wave exists

`legacy/frontend/lib/contracts/*` is a parallel source of truth: nine files
of zod schemas covering domains the backend does not yet model
(`access`, `calendar/DayStatusMap`, `chat`, `city`, `natal` deeper than
`NatalPayload`, `profile`, `today` deeper than `TodayPayload`).

The frontend whitelist (`grace/frontend.paths`) explicitly forbids
locally-redeclared payload types (W-2.0 Variant C, lint rule
`contracts-only-import`). Subsequent waves W-2.2..W-2.8 will pull files
out of `legacy/frontend/` and into `app/(grace)/**`,
`components/grace/**`, `lib/grace/**`. The instant a migrated file does
`import { TodayPayload } from "@/lib/contracts/today"`, ESLint Gate 8
fails (foreign import) and the drift gate (W-1.1B) is bypassed because
the local zod schema was never reconciled with `apps/api/app/schemas/`.

This wave reconciles the two sources **before** any feature TSX is
migrated. No UI behaviour, no route, no page, no component is touched
in this wave.

Two distinct concerns must be separated:

1. **Shape source of truth.** Pydantic schemas in `apps/api/app/schemas/`
   produce `packages/contracts/_generated.ts`. Drift gate (W-1.1B)
   guarantees byte-identity. Frontend imports types from
   `packages/contracts` only.

2. **Runtime validation.** The legacy frontend uses zod to parse mock
   payloads at boundary, both for early failure and for narrowing the
   discriminated unions used by UI. Pydantic is a server-side concern;
   it does not produce zod parsers. Either runtime validation is dropped
   on the frontend (rely on TS only) or it is reintroduced on top of
   `packages/contracts` types.

## Decision required from controller (record here, then proceed)

Pick exactly one option and write it under `## Decision` below.

- **A. Backend-driven, types-only.** Expand `apps/api/app/schemas/` to
  cover every shape used by legacy frontend (`AccessInfo`,
  `DayStatusMap`, `ChatMessage`, `ChatThread`, `City`, `ProfileMeta`,
  full nested `Today*` and `Natal*`). Regenerate `_generated.ts`.
  Frontend gets types only, no runtime validation in W-2.1. Pro: single
  source of truth, drift gate covers everything, lint rule stays strict.
  Con: backend Pydantic now defines fields the backend does not yet
  serve; runtime validation of mocks is lost until later wave reintro-
  duces zod on top.

- **B. Frontend-co-located zod, types pinned to backend.** Backend
  expands Pydantic only for shapes already emitted by handlers (today,
  health). For everything else, port `legacy/frontend/lib/contracts/*`
  to `lib/grace/contracts/*` (inside whitelist) as zod schemas whose
  inferred types **must** be assignable to/from
  `packages/contracts.*Payload` where overlap exists, checked by a new
  ESLint type-identity rule. Pro: keeps runtime validation, no premature
  Pydantic. Con: two SoTs in disjoint domains, type-identity check has
  to be written and is fragile.

- **C. Backend-driven shapes + frontend runtime guards.** Same Pydantic
  expansion as Option A. Add `lib/grace/validation/*` with zod schemas
  that are **constructed from** `packages/contracts` types (using
  `z.object` typed against `Pick<TodayPayload, ...>` etc., or a
  `satisfies ZodType<TodayPayload>` guard). Pro: drift gate covers
  shapes, runtime validation preserved, foreign imports stay forbidden.
  Con: most expensive; requires an explicit `satisfies ZodType<T>`
  pattern and a lint rule that flags `z.object(...)` without that
  pattern.

The remainder of this packet is conditional on the decision. Allowed
Write Scope below covers the union; worker uses only the subset matching
the chosen option.

## Escalation triggers

- A legacy zod schema cannot be expressed in Pydantic without changing
  field semantics (e.g. nullable Date with timezone). Stop, file an
  amendment to `apps/api/app/schemas/` that records the divergence as
  an explicit Pydantic field validator before continuing.
- The drift gate becomes red because `openapi-typescript` emits non-
  deterministic ordering for the new types. Fix the generator script,
  not the gate.
- Option B/C lint rule produces false positives on a Pydantic-emitted
  type. Narrow the rule's file glob; do not weaken the rule.
- Coverage diff (verification step 3) is non-empty and the missing
  domain is required by W-2.2 (Today). Stop, ship the missing schema
  before declaring W-2.1 done.
