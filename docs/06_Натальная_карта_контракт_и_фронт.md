# 06. Натальная карта: контракт и фронт

## Роль документа

Фиксирует архитектурную модель будущего разбора натальной карты в разделе **«Разборы»**.

Цель — backend может менять расчётный движок, структуру анализа и глубину отчёта без постоянного переписывания frontend.

## Главный принцип

Натальный разбор рендерится не как hardcoded экран, а как **версионированный документ из секций и блоков**.

Frontend — универсальный renderer, который получает контракт и отображает набор блочных примитивов.

## 1. Версионированный контракт

```ts
schemaVersion: "natal/v1"
```

Если меняется движок или структура отчёта:

1. инкрементируем версию схемы;
2. либо сохраняем текущую версию, если блоки и их `type` совместимы.

Главное правило: дискриминатор блока `type` должен сохраняться.

Frontend мапит:

```ts
block.type → component
```

Неизвестный тип блока не ломает экран: он пропускается, в dev/debug логируется.

## 2. Контент как `sections[].blocks[]`

```ts
export type NatalReadingPayload = {
  schemaVersion: "natal/v1"
  meta: NatalMeta
  highlights?: NatalHighlight[]
  spheres?: NatalSphere[]
  planets?: NatalPlanet[]
  sections: NatalSection[]
}

export type NatalSection = {
  id: string
  title: string
  subtitle?: string
  summary?: string
  blocks: NatalBlock[]
}
```

## 3. Блочные примитивы

Минимальный набор:

- `paragraph`
- `lead`
- `list`
- `callout`
- `pros_cons`
- `stat_grid`
- `heading`
- `divider`
- `spheres_widget`
- `planets_widget`

## 4. Типы блоков

```ts
export type ParagraphBlock = { type: "paragraph"; text: string }
export type LeadBlock = { type: "lead"; text: string }
export type HeadingBlock = { type: "heading"; text: string; level?: 2 | 3 | 4 }
export type ListBlock = { type: "list"; items: string[]; ordered?: boolean }

export type CalloutBlock = {
  type: "callout"
  tone?: "info" | "important" | "soft" | "warning"
  title?: string
  text: string
}

export type ProsConsBlock = {
  type: "pros_cons"
  prosTitle?: string
  consTitle?: string
  pros: string[]
  cons: string[]
}

export type StatGridBlock = {
  type: "stat_grid"
  items: Array<{ label: string; value: string; description?: string }>
}

export type DividerBlock = { type: "divider" }

export type SpheresWidgetBlock = {
  type: "spheres_widget"
  sphereIds?: string[]
  title?: string
}

export type PlanetsWidgetBlock = {
  type: "planets_widget"
  planetIds?: string[]
  title?: string
}
```

```ts
export type NatalBlock =
  | ParagraphBlock
  | LeadBlock
  | HeadingBlock
  | ListBlock
  | CalloutBlock
  | ProsConsBlock
  | StatGridBlock
  | DividerBlock
  | SpheresWidgetBlock
  | PlanetsWidgetBlock
```

## 5. Structured data отдельно от narrative

Structured data хранится отдельно:

- `meta`
- `highlights`
- `spheres`
- `planets`

Narrative живёт внутри `sections[].blocks[]`.

Это позволяет переиспользовать structured data на overview, вставлять виджеты внутри глав, менять структуру планет/сфер без переписывания narrative.


## 9. Planets

```ts
export type NatalPlanet = {
  id: string
  name: string
  sign?: string
  house?: number
  dignityScore?: number
  retrograde?: boolean
  summary: string
}
```

Если структура планет изменится, frontend правит только `planets_widget`, а не все главы.

## 10. Mobile-first навигация

Натальный разбор не должен быть одним огромным скроллом.

Фиксируем:

1. overview / TOC-страница;
2. отдельные страницы глав.

Overview route:

```text
/readings/natal
```

На overview:

- hero;
- короткое вступление;
- highlights;
- top spheres;
- список глав;
- возможно planets overview.

Section route:

```text
/readings/natal/[section]
```

Даёт нативный back в Telegram, sticky-header, scroll restoration, тестирование конкретной главы и меньше тяжёлого скролла.

## 11. Renderer frontend

```tsx
function RenderBlock({ block }: { block: NatalBlock }) {
  switch (block.type) {
    case "paragraph": return <ParagraphBlock block={block} />
    case "lead": return <LeadBlock block={block} />
    case "list": return <ListBlock block={block} />
    case "callout": return <CalloutBlock block={block} />
    case "pros_cons": return <ProsConsBlock block={block} />
    case "stat_grid": return <StatGridBlock block={block} />
    case "spheres_widget": return <SpheresWidget block={block} />
    case "planets_widget": return <PlanetsWidget block={block} />
    case "divider": return <DividerBlock />
    case "heading": return <HeadingBlock block={block} />
    default: return null
  }
}
```

## 12. Backend responsibility

Backend решает:

- какие главы есть;
- какие блоки внутри главы;
- где вставлять виджеты;
- какие highlights показать;
- какие top-сферы вывести.

Frontend только рендерит контракт.

## 13. Frontend responsibility

Frontend отвечает за:

- route structure;
- mobile-first layout;
- overview page;
- section page;
- block renderer;
- widgets;
- unknown block fallback;
- loading / error / locked states.

Frontend не выбирает главы, не переписывает narrative, не решает где вставить виджеты, не интерпретирует астрологию.

## 14. Locked / paywall state

Если натальный разбор закрыт:

- `/readings/natal` может показывать teaser;
- главы могут быть locked;
- CTA: подписка или приглашение друга.

Контракт должен поддерживать access state отдельно, аналогично TodayPayload.

## Главное решение

```text
versioned payload → sections[] → blocks[] → block renderer
```

Structured data живёт отдельно от narrative.

Навигация:

```text
/readings/natal → /readings/natal/[section]
```
