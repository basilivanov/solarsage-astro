# 06. Натальная карта: контракт и фронт

## Роль документа

Фиксирует архитектурную модель натального разбора в разделе «Разборы».
Актуальная версия контракта: **natal/v1**.

Цель — backend может менять расчётный движок, структуру анализа и глубину отчёта без переписывания frontend. Frontend — универсальный рендерер блочных примитивов.

Source of truth для типов: `lib/contracts/natal.ts` (Zod-схемы) и `packages/contracts/_generated.ts` (автогенерация из Pydantic). Этот документ описывает контракт; при расхождении между документом и кодом — код прав.

## 1. Версионированный контракт

```ts
schemaVersion: "natal/v1"
```

При изменении движка или структуры отчёта:

1. инкрементируем версию схемы, если блоки или их `type` несовместимы;
2. сохраняем текущую версию, если изменения обратно совместимы.

Главное правило: дискриминатор блока `type` должен сохраняться. Frontend мапит `block.type → component`. Неизвестный тип блока не ломает экран — он пропускается, в debug логируется.

## 2. Структура payload

```ts
type NatalReport = {
  schemaVersion: "natal/v1"
  meta: NatalMeta
  highlights?: Highlight[]
  spheres?: SphereScore[]
  planets?: PlanetScore[]
  sections: ReportSection[]
}
```

Narrative живёт внутри `sections[].blocks[]`. Structured data (`meta`, `highlights`, `spheres`, `planets`) живёт отдельно — переиспользуется на overview, вставляется в виджеты, меняется без переписывания narrative.

## 3. Meta

```ts
type NatalMeta = {
  name: string                          // Имя человека
  birth: {
    date: string                        // YYYY-MM-DD
    time?: string                        // HH:MM
    timezone?: string                    // IANA timezone
    place?: string                       // Город рождения
  }
  mode?: string                         // "western_only" и т.д.
  houseSystem?: string                  // "Placidus" и т.д.
  generatedAt?: string                  // ISO datetime
}
```

## 4. Highlights

Короткие бейджи для overview-хедера. Формат компактный: `label` — что, `value` — значение, `hint` — подсказка при тапе.

```ts
type Highlight = {
  id: string
  label: string       // "ASC", "Управитель", "Доминанта", "Дома"
  value: string       // "Дева", "Меркурий", "Карьера", "Placidus"
  hint?: string       // "восходящий знак", "финальный диспозитор"
}
```

Пример из реального разбора:
```json
[
  { "id": "asc", "label": "ASC", "value": "Дева", "hint": "восходящий знак" },
  { "id": "ruler", "label": "Управитель", "value": "Меркурий", "hint": "финальный диспозитор" },
  { "id": "dominant", "label": "Доминанта", "value": "Карьера", "hint": "статус и роль" },
  { "id": "house-system", "label": "Дома", "value": "Placidus" }
]
```

## 5. Spheres

Доминанты сфер жизни — рейтинг с числовыми оценками и тональностью.

```ts
type SphereScore = {
  id: string              // "career", "love", "money" и т.д.
  title: string           // "Карьера, статус, роль"
  dominance: number       // 0–5, суммарный вес
  metrics?: {
    involvement?: number  // вовлечённость
    ease?: number         // лёгкость
    tension?: number      // напряжение
    visibility?: number   // видимость
    shadow?: number       // тень
  }
  tone?: "neutral" | "strength" | "tension"
}
```

Пример:
```json
[
  { "id": "career", "title": "Карьера, статус, роль", "dominance": 3.65, "tone": "strength" },
  { "id": "self", "title": "Личность и вход в жизнь", "dominance": 3.29 },
  { "id": "work", "title": "Работа, режим, здоровье", "dominance": 0.84, "tone": "tension" }
]
```

## 6. Planets

Оценки планет — Composite Score, достоинство, безопасность и т.д.

```ts
type PlanetScore = {
  id: string              // "sun", "moon", "mercury" и т.д.
  name: string            // "Солнце", "Луна" (русские имена)
  sign: string            // "Дева", "Козерог"
  house?: number | string  // номер дома
  composite?: number       // Composite Score
  dignity?: number         // достоинство (Essential Dignity)
  bonification?: number    // бонус/проблема ( sect, combustion, etc. )
  inSect?: boolean         // в секте?
  dispositor?: string      // "Меркурий" — кто управляет
  note?: string            // "финальный диспозитор", "в изгнании"
}
```

Пример:
```json
{ "id": "mercury", "name": "Меркурий", "sign": "Дева", "house": 1, "composite": 8, "dignity": 9, "bonification": -2, "inSect": true, "dispositor": "Меркурий", "note": "финальный диспозитор" }
```

## 7. Sections и Blocks

```ts
type ReportSection = {
  id: string              // "portrait", "asc", "aspects" и т.д.
  title: string           // "Из чего ты собрана"
  eyebrow?: string        // "Глава 1"
  summary?: string        // Краткое описание для TOC
  blocks: Block[]
}
```

### 7.1. Блочные примитивы (полный набор)

```ts
type Block =
  | ParagraphBlock
  | LeadBlock
  | HeadingBlock
  | ListBlock
  | CalloutBlock
  | ProsConsBlock
  | StatGridBlock
  | QuoteBlock
  | DividerBlock
  | SpheresWidgetBlock
  | PlanetsWidgetBlock
```

#### ParagraphBlock
```ts
{ type: "paragraph", text: string }
```
Обычный текстовый абзац. Основной строительный блок для narrative.

#### LeadBlock
```ts
{ type: "lead", text: string }
```
Крупный абзац-вывод. Визуально выделен (крупнее шрифтом, жирнее). Используется для главных мыслей секции, врезок-резюме.

Пример из разбора: «Ты не входишь в жизнь хаотично. У тебя есть встроенный внутренний редактор.»

#### HeadingBlock
```ts
{ type: "heading", level: 2 | 3, text: string }
```
Подзаголовок внутри секции. level 2 — крупный, level 3 — мелкий.

Пример: `{ type: "heading", level: 2, text: "ASC в Деве" }`

#### ListBlock
```ts
{ type: "list", style?: "bullet" | "check", items: string[] }
```
Список. `bullet` — маркированный (точки), `check` — чек-лист (галочки).

Пример:
```json
{
  "type": "list",
  "style": "check",
  "items": [
    "сначала понять, потом довериться",
    "сначала проверить, потом сказать «да»"
  ]
}
```

#### CalloutBlock
```ts
{
  type: "callout"
  tone?: "neutral" | "strength" | "risk" | "insight"
  title?: string
  text: string
}
```
Выделенный блок с акцентом. Tone определяет визуальный стиль:
- `neutral` — нейтральная информация
- `strength` — сильная сторона карты (зелёный акцент)
- `risk` — предупреждение, зона напряжения (красный/оранжевый акцент)
- `insight` — ключевое озарение, главный вывод (фиолетовый акцент)

Пример из разбора:
```json
{
  "type": "callout",
  "tone": "insight",
  "title": "Главная архитектурная ось",
  "text": "Меркурий в Деве в 1 доме, в обители и экзальтации, плюс финальный диспозитор всей системы."
}
```

#### ProsConsBlock
```ts
{
  type: "pros_cons"
  prosLabel?: string     // по умолчанию "Сила" / "За"
  consLabel?: string     // по умолчанию "Риск" / "Против"
  pros?: string[]
  cons?: string[]
}
```
Блок «за/против». Используется для аспектов, сфер, планетарных функций.

Пример:
```json
{
  "type": "pros_cons",
  "pros": ["живая нервная система", "интуитивная скорость"],
  "cons": ["эмоциональные скачки", "трудность долго оставаться в связи без воздуха"]
}
```

#### StatGridBlock
```ts
{
  type: "stat_grid"
  items: Array<{ label: string, value: string }>
}
```
Сетка «ключ — значение». Используется для сводных данных.

#### QuoteBlock
```ts
{
  type: "quote"
  text: string
  cite?: string           // источник цитаты
}
```
Цитата. Визуально — стилизованный блок с отступом и кавычками.

#### DividerBlock
```ts
{ type: "divider" }
```
Горизонтальный разделитель между смысловыми блоками.

#### SpheresWidgetBlock
```ts
{
  type: "spheres_widget"
  limit?: number          // сколько сфер показать (default: все)
}
```
Виджет доминантных сфер. Использует данные из `spheres` в payload. Рендерится отдельным компонентом `SpheresWidget`.

#### PlanetsWidgetBlock
```ts
{
  type: "planets_widget"
  // planetIds?: string[] — пока не используется, показывает все
}
```
Виджет планетарных оценок. Использует данные из `planets` в payload. Рендерится `PlanetsWidget`.

## 8. Маппинг реального контента на блоки

Разбор Жанны (`zhanna_natal_friendly_v3.md`) — 531 строка текста. Ниже показано, как каждый тип контента мапится на блоки:

### Секции → ReportSection

| Секция в разборе | id | eyebrow | title |
|---|---|---|---|
| Главное впечатление | `overview` | Глава 1 | Из чего ты собрана |
| Асцендент + управители | `asc` | Глава 2 | Как ты входишь в жизнь |
| Солнце | `sun` | Глава 3 | Ядро личности |
| ... | ... | ... | ... |
| Аспекты | `aspects` | Глава N | Главные аспекты |
| Конфигурации | `configurations` | ... | ... |
| Итог | `synthesis` | Последняя | Итоговый синтез |

### Текст → блоки

| Контент в разборе | Блок |
|---|---|
| «У тебя карта не мягкого растворения, а умной собранности...» | `{ type: "lead", text: "..." }` |
| Обычный абзац текста | `{ type: "paragraph", text: "..." }` |
| «### Солнце соединение Уран» | `{ type: "heading", level: 2, text: "Солнце соединение Уран" }` |
| Список «- сначала понять», «- сначала проверить» | `{ type: "list", style: "check", items: [...] }` |
| Список «сильное считывание», «мягкий способ» | `{ type: "list", style: "bullet", items: [...] }` |
| Выделенный блок «Главный паттерн тени» | `{ type: "callout", tone: "risk", title: "Главный паттерн тени", text: "..." }` |
| Блок «Сила/Риск» для аспекта | `{ type: "pros_cons", pros: [...], cons: [...] }` |
| Разделитель между аспектами | `{ type: "divider" }` |
| Виджет доминантных сфер | `{ type: "spheres_widget" }` |
| Виджет планетарных оценок | `{ type: "planets_widget" }` |

## 9. Mobile-first навигация

Натальный разбор не должен быть одним огромным скроллом.

Структура:

1. **Overview / TOC-страница** (`/readings/natal`) — hero, highlights, top spheres, список глав;
2. **Отдельные страницы глав** (`/readings/natal/[id]`) — каждая глава на своём роуте.

Это даёт нативный back в Telegram, sticky-header, scroll restoration, тестирование конкретной главы и меньше тяжёлого скролла.

## 10. Renderer

```tsx
function RenderBlock({ block }: { block: Block }) {
  switch (block.type) {
    case "paragraph":      return <ParagraphBlock block={block} />
    case "lead":           return <LeadBlock block={block} />
    case "heading":        return <HeadingBlock block={block} />
    case "list":           return <ListBlock block={block} />
    case "callout":        return <CalloutBlock block={block} />
    case "pros_cons":      return <ProsConsBlock block={block} />
    case "stat_grid":      return <StatGridBlock block={block} />
    case "quote":          return <QuoteBlock block={block} />
    case "divider":        return <DividerBlock />
    case "spheres_widget": return <SpheresWidget block={block} />
    case "planets_widget": return <PlanetsWidget block={block} />
    default:               return null // unknown type — skip, log in debug
  }
}
```

## 12. Backend responsibility

Backend решает:
- какие главы есть и в каком порядке;
- какие блоки внутри главы;
- где вставлять виджеты (spheres_widget, planets_widget);
- какие highlights показать;
- какие top-сферы вывести;
- какие тональности callout (strength/risk/insight/neutral).

Frontend только рендерит контракт. Не выбирает главы, не переписывает narrative, не интерпретирует астрологию.

## 13. Frontend responsibility

Frontend отвечает за:
- route structure (`/readings/natal` → `/readings/natal/[id]`);
- mobile-first layout (max-w-md, sticky header);
- overview page (NatalToc);
- section page ( блок за блоком);
- block renderer (компоненты для каждого типа);
- widgets (SpheresWidget, PlanetsWidget);
- unknown block fallback (skip + debug log);
- loading / error / locked states;
- визуальные тональности callout (strength=risk/mp: нейтральный=серый).

## 14. Locked / paywall state

Если натальный разбор закрыт:
- `/readings/natal` показывает `LockedFeatureCard` с описанием и бейджем «Скоро»;
- главы могут быть locked;
- CTA: подписка или приглашение друга.

Контракт должен поддерживать access state отдельно, аналогично TodayPayload.

## 16. Таблица расхождений с предыдущей версией документа

Документ v0 (начальный) содержал ряд отличий от текущей реализации. Все изменения осознанные и зафиксированы в коде:

| Что | Deprecated v0 | Текущее (v1) | Причина |
|-----|---------------|-------------|---------|
| `CalloutBlock.tone` | `"info" \| "important" \| "soft" \| "warning"` | `"neutral" \| "strength" \| "risk" \| "insight"` | Астрологические тональности точнее, чем UI-термины |
| `ListBlock` | `items: string[], ordered?: boolean` | `style?: "bullet" \| "check", items: string[]` | bullet/check — визуально различимые стили; ordered не используется |
| `ProsConsBlock` titles | `prosTitle?, consTitle?` | `prosLabel?, consLabel?` | «label» точнее для UI-элемента |
| `ProsConsBlock` arrays | `pros: string[], cons: string[]` (required) | `pros?: string[], cons?: string[]` (optional) | Секции могут иметь только «за» или только «против» |
| `Highlight` | `{id, title, text, tone?}` | `{id, label, value, hint?}` | Компактные бейджи вместо карточек |
| `SphereScore` | `{id, title, score?, salience?, ease?, tension?, summary}` | `{id, title, dominance, metrics?: {...}, tone?}` | `dominance` — единый скор; `metrics` — детализация; `tone` — визуальный маркер |
| `PlanetScore` | `{id, name, sign?, house?, dignityScore?, retrograde?, summary}` | `{id, name, sign, house?, composite?, dignity?, bonification?, inSect?, dispositor?, note?}` | Composite Score вместо单一的 dignity; добавлены bonification, sect |
| `NatalMeta` | `{title, subtitle?, generatedAt, person?: {name?, birthDate?, birthTime?, birthPlace?}}` | `{name, birth: {date, time?, timezone?, place?}, mode?, houseSystem?, generatedAt?}` | Структура birth — объект с IANA timezone вместо плоских полей |
| `Section.subtitle` | `subtitle?: string` | `eyebrow?: string` | «eyebrow» — стандартный UI-термин для надзаголовка |
| `QuoteBlock` | Не описан | `{type: "quote", text, cite?}` | Добавлен по итогам mock-данных |

## Главное решение

```
versioned payload → sections[] → blocks[] → block renderer
```

Structured data живёт отдельно от narrative. Навигация:

```
/readings/natal → /readings/natal/[id]
```
