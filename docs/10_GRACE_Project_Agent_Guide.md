# 10. GRACE / Project Agent Guide

## Что это

Главный onboarding-документ для агента, который будет работать с проектом в новом чате.

Перед любыми изменениями агент должен прочитать:

1. `grace/README.md` — актуальная карта проекта, текущие модули, гейты
2. `09_Project_transfer_context.md`
3. `00_Обзор_продукта.md`
4. релевантный документ по зоне задачи

---

# Что строим

Telegram Mini App с персональным астрологическим ассистентом.

Продукт не должен звучать как общий гороскоп. Он должен ощущаться как персональный навигатор дня и периода.

Главная ценность: **попадание в личное состояние пользователя через глубокий расчёт и понятную интерпретацию**.

---

# Главный принцип

Всегда отвечать на вопрос:

**«Как это проявляется у меня?»**

Не достаточно сказать:

> Меркурий ретрограден.

Нужно объяснить:

> как это затрагивает пользователя;
> через какие дома и сферы;
> почему сегодня;
> что усиливает;
> что смягчает;
> что делать practically.

---

# Что нельзя ломать

## Продуктовые решения

- Today — главный экран MVP.
- Today = экран выбранного дня.
- Route: `/day/:date`.
- Верхние флаги раскрываются inline, не на отдельном экране.
- «Почему так у меня» — inline expansion, не отдельный экран.
- Календарь — навигация по дням, не месячный прогноз.
- 7-дневная лента использует 3 статуса: поддерживающий / ровный / напряжённый.
- Нет постоянного free-tier.
- Рефералка: пригласи друга — оба получают 14 дней.
- Сначала тратятся бонусные дни, потом подписка.

## Архитектурные решения

- Фронт не интерпретирует астрологию.
- Фронт не получает raw SolarSage.
- Backend отдаёт стабильный payload.
- LLM не получает весь raw JSON.
- LLM пишет по SemanticLayer.
- Все расчётные факторы нормализуются в AstroSignal.
- Считаем максимум, показываем отобранное.

Визуально:

- premium;

# Работа с фронтом

Если задача по фронту:

1. Сначала проверить `packages/contracts/index.ts` — текущие фронтовые типы.
2. Если нужны backend-схемы — `apps/api/app/schemas/*` — source of truth для API контрактов.
3. Идентифицировать затронутые модули перед редактированием (`grace/knowledge-graph.xml`).
2. Не менять UI-каркас без причины.
3. Выровнять types / fixtures / schemas / adapters с контрактами.
4. Не добавлять астрологическую логику на фронт.
5. Unknown block types должны не ломать экран.
6. Для API-compatible data не использовать React-компоненты внутри JSON.
7. Для backend JSON использовать ISO strings, не Date objects.

---

# Работа с backend

Если задача по backend:

1. Главный endpoint: `GET /api/day/:date`.
2. Начинать с access/profile/cache pipeline.
3. SolarSage raw хранить отдельно.
4. Нормализацию делать через AstroSignal.
5. Scoring должен учитывать support, friction, intensity, phase, totalSalience.
6. TodayPayload кэшировать отдельно от raw/semantic.
7. Хранить версии calculation/normalization/scoring/prompt/contract.
8. Не отдавать frontend raw SolarSage.

---

# Работа с LLM

Если задача по LLM:

1. LLM не придумывает расчёты.
2. LLM получает curated SemanticLayer.
3. Ответ LLM должен быть strict JSON под контракт.
4. LLM не упоминает internal scoring fields.
5. В тексте Today меньше терминов, в «Почему так у меня» больше астрологии.
6. Главный стиль why-блока смотри в `03_Почему_так_у_меня.md`.

---

# Работа с документацией

Если агент меняет важное решение:

1. Явно указать, какой документ нужно обновить.
2. Не менять контракт молча.
3. Если меняется фронтовый payload — обновить `packages/contracts/` и перегенерировать контракты.
4. legacy/ физически удалён — не использовать legacy migration docs (`grace/packets/archive/`) как активный план.
4. Если меняется расчётная модель — обновить `04_SolarSage_нормализация_скоринг_кеширование.md`.
5. Если меняется экран — обновить соответствующий UX-документ.

# Что уточнять у владельца проекта

Уточнять, если задача касается:

- добавления 5-й вкладки «Спросить» в MVP;
- изменения монетизации;
- изменения количества free/trial дней;
- изменения статусов 7-дневной ленты;
- изменения структуры Today;
- изменения контракта `TodayPayload`;
- добавления новых paid-разделов.

# Короткий elevator summary для агента

Мы строим Telegram Mini App, где главный экран — персональный Today-разбор на основе глубоких астрологических расчётов. Backend считает максимум через SolarSage и дополнительные техники, нормализует всё в AstroSignal, скорит день, собирает SemanticLayer, отдаёт LLM curated context, кэширует TodayPayload. Фронт рендерит стабильный payload и не знает расчётную кухню. Главный UX-принцип: пользователь должен понимать не «что на небе», а «как это проявляется у меня».
