# TZ: W-PROFILE-GENDER-ONBOARDING

Date: 2026-06-09
Status: ready for coder
Scope: onboarding, profile model, prompt context, gendered Russian wording

## 1. Goal

Add required gender selection during onboarding so Russian user-facing texts can use the correct masculine or feminine grammatical forms.

The product writes personal long-form texts on `ты`. In Russian this still requires grammatical gender in many phrases:

- `ты готов` / `ты готова`
- `ты почувствовал` / `ты почувствовала`
- `ты родился` / `ты родилась`

Without this field the output can feel impersonal or wrong.

## 2. Product decision

Use binary selection only:

- `male` — мужчина, write in masculine forms;
- `female` — женщина, write in feminine forms.

No neutral option in this packet.

UI copy on onboarding:

`Ты мужчина или женщина?`

Buttons/options:

- `Мужчина`
- `Женщина`

## 3. Data model

Add field to user profile:

`gender: "male" | "female"`

Recommended DB column:

- table: `user_profiles`
- column: `gender`
- type: string/varchar or enum-compatible string
- allowed values: `male`, `female`

Migration behavior:

- Add column as nullable first for existing rows.
- New onboarding must require it.
- Existing profiles with `gender IS NULL` must be treated as incomplete and asked to complete the field before generating new personalized readings.
- Do not silently default existing users to male or female.

This follows `docs/FAILURE_HANDLING_CANON.md`: no hidden default that can produce a wrong user-facing result.

## 4. API contract

Update profile create/update/read schemas to include gender.

Input validation:

- Accept only `male` or `female`.
- Reject missing gender on profile creation/onboarding submit.
- Reject any other value with validation error.

Profile read response should expose gender so the frontend can render the selected value in profile edit.

## 5. Onboarding UI

Add gender step/field to onboarding.

Requirements:

- User cannot finish onboarding without choosing male/female.
- Selection must be obvious and mobile-friendly.
- Selected state must be visually clear.
- Existing users with missing gender should see a profile completion prompt/step.

Suggested placement:

- Near name/birth details, before final submit.

## 6. Profile edit UI

Add gender editing in profile settings.

Requirements:

- User can change male/female after onboarding.
- Saving gender invalidates generated text caches that depend on prompt wording.
- Do not require recalculation of raw astrology if only gender changes; invalidate only semantic/LLM/user-facing payload layers where applicable.

## 7. Prompt context

All LLM prompts that produce Russian user-facing text must receive gender context.

Prompt instruction examples:

For `male`:

`Пиши на «ты», в мужском роде. Используй формы: готов, почувствовал, родился.`

For `female`:

`Пиши на «ты», в женском роде. Используй формы: готова, почувствовала, родилась.`

Apply to at least:

- Today interpretation;
- Today important details;
- Why sections;
- Natal reading;
- Horary answer;
- any future readings generated by LLM.

## 8. Backend/cache behavior

Gender affects wording, not astrology calculation.

Rules:

- Do not include gender in raw SolarSage calculation inputs.
- Do include gender in semantic/LLM prompt input hash where text output depends on it.
- If gender changes, invalidate cached generated text/payloads that contain gendered wording.
- Raw natal/period/daily chart snapshots may remain valid.

## 9. Tests

Backend tests:

1. Profile create without gender fails validation.
2. Profile create with `male` succeeds.
3. Profile create with `female` succeeds.
4. Invalid gender value fails validation.
5. Existing profile with missing gender is treated as incomplete for new personalized reading generation.
6. Gender is included in prompt input hash for LLM-generated text layers.
7. Updating gender invalidates user-facing generated text caches but does not delete raw chart snapshots.

Frontend tests:

1. Onboarding shows `Мужчина` and `Женщина` options.
2. Submit disabled or blocked until one option is selected.
3. Selected option is visibly active.
4. Profile edit can change gender and save.

Prompt/guard tests:

1. For male profile, prompt contains masculine instruction.
2. For female profile, prompt contains feminine instruction.
3. Generated fixture/sample for male must not contain common feminine-only forms where the sentence refers to the user.
4. Generated fixture/sample for female must not contain common masculine-only forms where the sentence refers to the user.

## 10. Acceptance criteria

- New users must select `Мужчина` or `Женщина` during onboarding.
- Profile API persists and returns `gender`.
- LLM prompts receive gender-specific Russian wording instruction.
- Existing users without gender are asked to complete profile instead of receiving silently defaulted wording.
- Gender change invalidates generated user-facing text layers.
- Raw astrology calculation remains independent of gender.
- Tests cover validation, onboarding UI, prompt context, and cache invalidation.

## 11. Out of scope

- No non-binary/neutral option in this packet.
- Do not use gender for astrology calculation.
- Do not add gender-based interpretations like “male energy” or “female energy”.
- Do not redesign onboarding beyond the required gender field.
- Do not change payment/subscription logic.
