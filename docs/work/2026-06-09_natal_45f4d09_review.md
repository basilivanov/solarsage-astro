This follow-up targets the remaining blocker from the previous review: after removing the silent `female` fallback, the onboarding save path still allowed nullable `state.gender` to be used where the `Profile` contract requires `male|female`.

## Findings

### B1 — explicit finish guard added

Status: ACCEPTED

`finish()` now stops before profile construction and persistence when gender is missing:

```ts
if (!state.gender) {
  dispatch({ type: "go_to_step", value: "gender" })
  return
}
```

This prevents both:

- `saveProfile(profile)` with invalid `gender: null`;
- `updateProfile({ gender: null })`.

### B2 — gender type is narrowed before constructing Profile

Status: ACCEPTED

After the guard, the code narrows the value:

```ts
const gender: "male" | "female" = state.gender
```

The narrowed `gender` is then used in the local `Profile` object and in the backend update payload. This resolves the TypeScript-level issue raised in the previous review.

### B3 — reducer supports explicit go-to-step

Status: ACCEPTED

`OnboardingEvent` now includes:

```ts
{ type: "go_to_step"; value: StepKey }
```

and the reducer applies it through the known `STEP_ORDER` list. This is enough for the current guard path.

### B4 — fixture update

Status: ACCEPTED

`useChat.test.ts` profile fixture now includes `gender: 'male'`, which aligns it with the profile contract.
